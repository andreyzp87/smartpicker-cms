# Export Implementation Reference

This document describes how SmartPicker CMS builds and serves the public export payloads consumed by the Astro frontend.

The implementation source of truth is:

- `src/services/export.service.ts`
- `src/index.ts`
- `src/storage/*`
- `src/routes/exports.ts`
- `src/cli.ts`
- `src/db/schema.ts`

The generated TypeScript contract written to `types.ts` is emitted by `ExportService.buildTypesExportFile()` and is the exact field-level contract for frontend consumers.

## Scope

This reference covers:

- runtime entry points
- storage and serving behavior
- export families and file layout
- snapshot generation and filtering rules
- derived compatibility logic
- limited export selection
- aggregate payload generation
- source database schema used by exports

## Runtime Entry Points

Exports are generated in three ways:

- `pnpm cli export --type <type> [--limited]` via `src/cli.ts`
- protected tRPC mutations under `exports.*` via `src/routes/exports.ts`
- direct `ExportService` calls from application code

Static export files are served by the Hono app from `/api/exports/*` via `src/index.ts`.

Important implementation detail:

- The Hono static route always serves `./data/exports`.
- The storage layer can instead be configured to write to S3.
- In local storage mode, `/api/exports/*` matches the generated files.
- In S3 mode, `StorageDriver.write()` returns S3 or CDN-style URLs, but the app still statically serves `./data/exports`, so `/api/exports/*` is only guaranteed when local storage is used.

## Storage Behavior

Storage is selected by `STORAGE_TYPE` in `src/storage/index.ts`.

### Local storage

Defaults:

- `EXPORTS_PATH=./data/exports`
- `EXPORTS_URL_BASE=/api/exports`

Behavior:

- JSON is pretty-printed with two-space indentation.
- Writes are atomic: data is written to a temp file and renamed into place.
- Public URLs are generated under `/api/exports/...`.

### S3 storage

Required:

- `STORAGE_TYPE=s3`
- `S3_BUCKET`

Optional:

- `S3_REGION`
- `S3_ACCESS_KEY_ID`
- `S3_SECRET_ACCESS_KEY`
- `S3_ENDPOINT`
- `S3_PUBLIC_URL_BASE`

Behavior:

- Objects are uploaded directly to the bucket.
- Returned URLs come from `S3_PUBLIC_URL_BASE`, the endpoint, or the default AWS S3 URL shape.

- `protocols`
- `integrations`
- `platforms`
- `hubs`
- `catalog`

## Generation Lifecycle

### Full export

`generateAllExports()`:

1. builds one shared snapshot
2. writes all export families in parallel
3. regenerates `types.ts` as part of the products export
4. triggers `CLOUDFLARE_DEPLOY_HOOK` if configured

Important implementation detail:

- The deploy hook is triggered only by `generateAllExports()`.
- Running a single export family does not trigger the hook.

### Single-family export

Each `generate*Export()` method:

- builds its own snapshot unless one is passed in
- writes only its own family
- deletes stale detail JSON files for that family

Important implementation detail:

- Product detail files are written only when `writeProductDetails` is enabled.
- `generateProductsExport()` and `generateAllExports()` enable product detail writes.
- Other export families do not rewrite product detail files.

### Stale file cleanup

Before writing detail families, the service deletes stale `.json` files not present in the current snapshot:

- `products/*`
- `manufacturers/*`
- `categories/*`
- `integrations/*`
- `platforms/*`
- `hubs/*`
- `protocols/*`

Preserved index files:

- `products/slugs.json`
- `manufacturers/slugs.json`
- `categories/paths.json`
- `integrations/slugs.json`
- `platforms/slugs.json`
- `hubs/slugs.json`
- `protocols/slugs.json`

## Snapshot Build Rules

`buildExportSnapshot()` is the central export pipeline.

It loads:

- all manufacturers
- all categories
- published integrations
- published platforms
- published hubs
- published products, processed in batches

Products are processed in batches of `250`.

Sorting behavior:

- product summaries: name
- product slugs: slug
- manufacturers: name
- integrations: name
- platforms: name
- hubs: name
- categories: path
- protocols: declaration order from `PROTOCOL_DEFINITIONS`
- most detail slug lists: sorted by referenced product name or slug as noted below

## Public Inclusion Rules

### Entity status filters

- Only products with `status = published` are exported.
- Only integrations with `status = published` are used.
- Only platforms with `status = published` are used.
- Only hubs with `status = published` are used.
- Manufacturers are exported only if at least one exported product belongs to them.
- Categories are exported only if `deviceCount > 0` in full mode.
- Categories keep the full tree in limited mode, even when a category has `deviceCount = 0`.

### Compatibility visibility filters

- Compatibility rows are public only when `reviewState = approved`.
- Positive compatibility statuses are `verified`, `supported`, and `reported`.
- Non-positive approved rows can still appear in product detail compatibility lists.
- Pending and rejected compatibility rows are never exported.

### Protocol export scope

The protocol family is fixed to:

- `zigbee`
- `zwave`
- `matter`
- `wifi`
- `thread`
- `bluetooth`

Important implementation detail:

- `products.primaryProtocol` and `integrations.primaryProtocol` may still contain database enum values like `proprietary` or `multi`.
- Those values can appear in product payloads and catalog protocol facets.
- They do not get dedicated entries in `protocols.json` or `site.protocolCounts`.

## Derived Compatibility Logic

Compatibility is built per product in `buildPublicCompatibility()`.

### Product to integration rows

Included when:

- compatibility row is approved
- referenced integration is published

Exported fields:

- `integration`
- `manufacturer`
- `platformSlugs`
- `status`
- `reviewState`
- `supportSummary`
- `canonicalSource`
- `lastConfirmedAt`

Important implementation detail:

- `platformSlugs` is derived from published `platform_integrations` links for the referenced integration.
- `firstSeenAt`, `internalNotes`, and raw evidence rows are not exported.

### Product to hub rows

Included when:

- compatibility row is approved
- referenced hub is published

Exported fields:

- `hub`
- `manufacturer`
- `status`
- `reviewState`
- `supportSummary`
- `canonicalSource`
- `lastConfirmedAt`

### Derived product to platform rows

`platformCompatibility` is not stored directly in the database.

It is derived from:

- approved positive product-to-integration rows
- published integration-to-platform relationships

Rules:

- only positive integration compatibility rows participate
- each platform appears once
- `status` is the strongest contributing status by rank
- `viaIntegrations` lists the integrations that produced the platform match

Status rank:

1. `verified`
2. `supported`
3. `reported`
4. `untested`
5. `incompatible`

Implementation note:

- The service compares status ranks numerically and keeps the higher-ranked positive status when multiple integrations map to the same platform.

### Infrastructure hints

`infrastructureHints` on product detail are derived from:

- approved positive product-to-integration compatibility
- published integration hardware support links
- only infrastructure products (`productRole = infrastructure`)
- only published infrastructure products

Hints are grouped by:

- infrastructure product slug
- `requirementType`

Each hint includes:

- referenced infrastructure product
- manufacturer and category refs for that infrastructure product
- `requirementType`
- `viaIntegrations`
- `note`

## Per-Family Derivation Rules

### Products

`ProductExportSummary` is built from the product row plus derived compatibility and category metadata.

`searchText` is a lowercase, deduplicated concatenation of:

- product name
- model
- manufacturer name
- category name
- ancestor category names
- primary protocol
- compatible integration names
- compatible platform names
- compatible hub names

`compatibilityStatuses` is the sorted union of statuses across exported integration, platform, and hub compatibility rows.

`ProductExport` extends the summary with:

- `description`
- optional `zigbeeDetails`
- optional `zwaveDetails`
- `integrationCompatibility`
- `platformCompatibility`
- `hubCompatibility`
- `infrastructureHints`

Important implementation detail:

- `integrationCompatibility` includes approved rows even when status is non-positive.
- `hubCompatibility` includes approved rows even when status is non-positive.
- `compatibleIntegrationSlugs`, `compatiblePlatformSlugs`, and `compatibleHubSlugs` include only positive rows.

### Manufacturers

A manufacturer aggregate is built from exported products only.

Derived fields:

- `deviceCount`: number of exported products for the manufacturer
- `compatibilityCounts.integrations`: distinct positive integration slugs across the manufacturer’s exported products
- `compatibilityCounts.platforms`: distinct positive platform slugs across the manufacturer’s exported products
- `compatibilityCounts.hubs`: distinct positive hub slugs across the manufacturer’s exported products
- `updatedAt`: max `product.updatedAt` across exported products, not `manufacturer.updatedAt`

Detail-only fields:

- `deviceSlugs`
- `integrationSlugs`
- `platformSlugs`
- `hubSlugs`

### Categories

Category metadata is derived from the full category tree.

Derived fields:

- `path`: slash-joined slug path
- `pathSegments`
- `ancestors`
- `children`: sorted by `sortOrder`, then `name`
- `directDeviceCount`
- `deviceCount`
- `updatedAt`: max `product.updatedAt` across the subtree

Aggregation behavior:

- direct device counts come from products directly in the category
- subtree device counts include descendants
- ancestor categories are incremented when a product belongs to a descendant category

### Integrations

Integration aggregates are built from:

- published integrations
- positive approved product compatibility
- published platform links
- infrastructure support links

Derived fields:

- `platformSlugs`: published linked platforms
- `compatibleDeviceCount`: number of exported products with positive approved compatibility to the integration
- `statusBreakdown`: counts of positive approved compatibility statuses
- `searchText`: name, manufacturer, primary protocol, description, and linked platform names
- `updatedAt`: max of `integration.updatedAt` and latest compatible product update time
- `hardwareHints`: infrastructure hardware links for published infrastructure products

Important implementation detail:

- `hardwareHints` on the integration export come directly from `integration_hardware_support`.
- `infrastructureHints` on the product export are a separate derived view grouped through the product’s compatible integrations.

### Platforms

Platform aggregates are built from:

- published platforms
- published integration links
- derived platform compatibility from positive approved product-to-integration rows

Derived fields:

- `integrationSlugs`: published linked integrations
- `compatibleDeviceCountDerived`: number of products deriving compatibility through integrations
- `statusBreakdown`: counts of derived platform statuses
- `searchText`: name, manufacturer, kind, description, linked integration names
- `updatedAt`: max of `platform.updatedAt` and latest contributing compatible product update time

### Hubs

Hub aggregates are built from:

- published hubs
- approved product-to-hub compatibility rows

Derived fields:

- `compatibleDeviceCount`: number of exported products with positive approved hub compatibility
- `statusBreakdown`: counts of positive approved compatibility statuses
- `searchText`: name, manufacturer, description
- `updatedAt`: max of `hub.updatedAt` and latest contributing compatible product update time

### Protocols

Protocols come from the fixed in-code definitions, not directly from database rows:

- `zigbee`
- `zwave`
- `matter`
- `wifi`
- `thread`
- `bluetooth`

Derived fields are based on exported products whose `primaryProtocol` matches one of those six values.

Counts:

- `deviceCount`
- `localControlCount`
- `cloudDependentCount`
- `matterCertifiedCount`
- `integrationCount`
- `platformCount`
- `hubCount`

Important implementation detail:

- `integrationCount`, `platformCount`, and `hubCount` are not counts of entities whose own primary protocol matches the protocol.
- They are counts of distinct compatible integrations, derived platforms, and compatible hubs found on products assigned to that protocol.

## Limited Export Mode

Limited mode keeps the same payload schemas but restricts which products are exported.

Selection algorithm:

1. load all published product candidates
2. count published products per manufacturer
3. keep the top `20` manufacturers by product count
4. for those manufacturers, count all compatibility references per product across both compatibility tables
5. rank products by:
   - higher compatibility reference count
   - newer `updatedAt`
   - name
   - id
6. keep up to `10` products per `manufacturer + category + primaryProtocol` combination

Combo key details:

- null category uses `__uncategorized__`
- null protocol uses `__unknown_protocol__`

Important implementation detail:

- The limit is per manufacturer, category, and primary protocol combination.
- It is not a flat per-manufacturer cap.
- Integrations, platforms, hubs, and protocols remain published/full-scope.
- Categories remain present even if they end up empty in limited mode.

## Aggregate Payload Logic

### Catalog

`catalog.json` contains:

- the full exported product summary list
- facet counts derived from that list

Facet logic:

- `protocols`: counts `product.primaryProtocol` values exactly as present on exported products
- `manufacturers`: counts product manufacturer refs
- `categories`: counts the direct category path from `product.categoryPath`
- `integrations`: counts positive compatible integration slugs
- `platforms`: counts positive compatible platform slugs
- `hubs`: counts positive compatible hub slugs
- `features`: counts boolean product flags

Important implementation detail:

- Catalog protocol facets may include `proprietary` and `multi`.
- `protocols.json` does not.

### Search

`search.json` contains static search items for:

- devices
- manufacturers
- integrations
- platforms
- hubs

Entity URLs:

- product: `/devices/{slug}`
- manufacturer: `/manufacturers/{slug}`
- integration: `/integrations/{slug}`
- platform: `/platforms/{slug}`
- hub: `/hubs/{slug}`

### Site

`site.json` contains:

- aggregate entity counts
- protocol counts for the fixed public protocol family
- homepage-style featured slug lists

Featured list logic:

- `featuredDeviceSlugs`: 12 most recently updated exported products
- `recentlyUpdatedDeviceSlugs`: 24 most recently updated exported products
- `featuredIntegrationSlugs`: top 12 integrations by `compatibleDeviceCount`, then name
- `featuredPlatformSlugs`: top 12 platforms by `compatibleDeviceCountDerived`, then name
- `featuredHubSlugs`: top 12 hubs by `compatibleDeviceCount`, then name

### Sitemap

`sitemap.json` is a wrapped payload, not a raw array.

Included pages:

- static routes: `/`, `/about`, `/devices`, `/integrations`, `/platforms`, `/hubs`, `/manufacturers`, `/categories`, `/protocols`
- all exported product detail pages
- all exported integration detail pages
- all exported platform detail pages
- all exported hub detail pages
- all exported manufacturer detail pages
- all exported category detail pages
- all exported protocol detail pages

`lastModified` rules:

- products, integrations, platforms, hubs: entity `updatedAt`
- manufacturers: aggregate `updatedAt` or snapshot `generated`
- categories: aggregate `updatedAt` or snapshot `generated`
- protocols: aggregate `updatedAt` or snapshot `generated`
- static routes: snapshot `generated`

## Astro Usage Notes

- `/devices` can read `products.json` or `catalog.json` depending on whether facet data is needed.
- `/devices/[slug]` uses `products/{slug}.json`.
- `/manufacturers` and `/manufacturers/[slug]` use the manufacturers family.
- `/categories` and `/categories/[...path]` use the categories family.
- `/protocols` and `/protocols/[slug]` use the protocols family.
- `/integrations` and `/integrations/[slug]` use the integrations family.
- `/platforms` and `/platforms/[slug]` use the platforms family.
- `/hubs` and `/hubs/[slug]` use the hubs family.
- `search.json` is the static search index for devices, manufacturers, integrations, platforms, and hubs.
- `site.json` contains homepage and global aggregate data.
- `sitemap.json` is a wrapped payload with `urls`, not a raw array.
- `types.ts` is the generated frontend contract and should be treated as exact.
