# Export Data Contract

This document describes the JSON files generated by SmartPicker CMS for the Astro frontend.

`src/services/export.service.ts` is the implementation source of truth. The service also generates `types.ts`, which is the exact TypeScript contract consumed by the frontend.

## Overview

Exports are served from `/api/exports/*`.

When local file storage is enabled, the generated files are written to `data/exports/`.

Current export families:

- products
- manufacturers
- categories
- protocols
- integrations
- platforms
- hubs
- catalog
- search
- site
- sitemap
- types

Note: the public export still uses the `hubs` filename family even though the CMS model is `commercialHubs`.

## Contract Version

Current version: `4.0`

Every JSON payload includes at least:

```json
{
  "generated": "2026-03-23T18:24:50.373Z",
  "version": "4.0"
}
```

## Export Rules

- Full export mode is the default.
- Limited export mode keeps the same schemas, but limits products to the top 20 manufacturers and up to 10 published devices per category/protocol combination.
- Only products with `status = published` are exported.
- Only integrations, platforms, and hubs with `status = published` are exported.
- Manufacturers are exported only if they have at least one exported product.
- Categories are exported with `deviceCount > 0` in full mode. In limited mode, the full category tree is kept.
- Protocol exports are always generated from the fixed protocol set: `zigbee`, `zwave`, `matter`, `wifi`, `thread`, `bluetooth`.
- Product compatibility rows are included only when `reviewState = approved`.
- Compatibility is considered positive only when `status` is one of `verified`, `supported`, or `reported`.
- `compatibleIntegrationSlugs`, `compatiblePlatformSlugs`, `compatibleHubSlugs`, derived platform compatibility, infrastructure hints, status breakdowns, and compatible device counts all use only positive compatibility rows.
- Product detail still includes approved non-positive integration and hub rows in `integrationCompatibility` and `hubCompatibility`.
- `platformCompatibility` is derived from approved positive integration compatibility rows and published integration-platform relationships.
- `types.ts` is regenerated when the products export runs, including `export all`.

## Astro Usage Notes

- `/devices` can use `products.json` or `catalog.json`, depending on whether facet data is needed.
- `/devices/[slug]` should use `products/{slug}.json`.
- `/manufacturers` and `/manufacturers/[slug]` use the manufacturers family.
- `/categories` and `/categories/[...path]` use the categories family.
- `/protocols` and `/protocols/[slug]` use the protocols family.
- `/integrations` and `/integrations/[slug]` use the integrations family.
- `/platforms` and `/platforms/[slug]` use the platforms family.
- `/hubs` and `/hubs/[slug]` use the hubs family.
- `search.json` is the static search index for devices, manufacturers, integrations, platforms, and hubs.
- `site.json` contains homepage and global aggregate data.
- `sitemap.json` is a wrapped payload with `urls`, not a raw array.
- `types.ts` is the exact generated TypeScript contract for the frontend.
